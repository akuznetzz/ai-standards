# Project-Specific AI Rules

Англоязычный оригинал: [project-rules.md](project-rules.md)

## Политика языка документации

- Англоязычные файлы остаются source of truth для поддерживаемой документации проекта.
- Для `README.md` и каждого поддерживаемого файла под `docs/` нужно поддерживать русский локализованный эквивалент.
- Для русских локализованных эквивалентов используйте суффикс `.ru.md`.
- Если меняется английский оригинал или его русская локализация, обновляйте обе версии в одном наборе изменений.
- Если под `docs/` добавляется новый поддерживаемый документ, в той же задаче добавляйте и его русскую локализацию, если пользователь явно не одобрил исключение.
- По возможности сохраняйте ссылки между английскими оригиналами и русскими локализованными версиями.
- В ссылках на файлы репозитория внутри поддерживаемой документации используйте только относительные пути репозитория.
- Не используйте абсолютные пути локальной файловой системы вроде `/home/...` в ссылках документации репозитория.
- Не создавайте русские или английские эквиваленты для файлов под `docs/`, чьи имена содержат `-log-`.
- Считайте `docs/*-log-*.md` экспортами чатов, которые должны оставаться в исходном языке и исходной форме.

## Область действия документационной политики

- Это правило синхронизации применяется к `README.md` и поддерживаемой документации в `docs/`.
- Файлы под `docs/`, содержащие `-log-` в имени, исключены из политики двуязычной синхронизации.
- Шаблоны, сгенерированные файлы и исторические артефакты вне `docs/` не входят в область действия автоматически, если пользователь отдельно не попросил локализацию.

## Workflow

- Перед изменением документации проверяйте, не требует ли обновления также связанная английская или русская версия.
- При создании новой пары документов предпочитайте `name.md` для английской версии и `name.ru.md` для русской.
- Не переписывайте, не переводите и не парьте chat export files, в чьих именах содержится `-log-`.

## Release Workflow

- Release metadata репозитория хранится в `pyproject.toml` в полях `tool.ai-standards.version` и `tool.ai-standards.release_date`.
- `project.version` в `pyproject.toml` описывает Python support tooling package, а не canonical release version репозитория `ai-standards`.
- Меняйте `project.version` только тогда, когда сам Python support package действительно требует package-version bump, например из-за изменений скриптов или зависимостей.
- Метаданные self-hosted манифеста хранятся в `ai.project.toml` в полях `ai_standards_version`, `project_version` и `project_release_date`.
- Для preview, save и tag операций над release version в этом репозитории используйте `scripts/bump_version.py`.
- `scripts/bump_version.py save` разрешён только на чистом git worktree.
- `scripts/bump_version.py tag` разрешён только на чистом git worktree и только из ветки `main`.
- Создание релизного тега остаётся отдельным шагом по отношению к сохранению версии и git commit.
