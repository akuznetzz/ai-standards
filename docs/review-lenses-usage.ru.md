# Руководство по применению review-lenses

Англоязычный оригинал: [review-lenses-usage.md](review-lenses-usage.md)

Это руководство объясняет, как применять возможность `review-lenses` из `ai-standards` в реальных сценариях разработки.

Эта возможность задаёт одну переиспользуемую модель проверки с тремя ракурсами:

- `Reuse`
- `Quality`
- `Efficiency`

Она предназначена для очистки и структурного улучшения недавних изменений без изменения поведения. Это не режим по умолчанию для каждой задачи и не замена тестам, линтерам или обычной проверке кода.

## Цели

Используйте `review-lenses`, когда нужно, чтобы агент сделал одно или оба из следующих действий:

- провёл проверку изменения по устойчивой методике нескольких ракурсов
- выполнил безопасную очистку и структурные улучшения без изменения ожидаемого поведения

Типичные результаты:

- меньше дублирования
- более ясный поток выполнения
- меньше шумных комментариев и отладочных остатков
- лучшее использование существующих проектных примитивов
- более конкретная проверка вместо расплывчатого “looks good / maybe refactor”

## Два режима

`review-lenses` следует использовать в двух явных режимах.

### 1. Review-Only

В режиме `review-only` агент анализирует целевой набор изменений или файлы и сообщает замечания, но не редактирует код.

Используйте этот режим, когда нужно:

- запускать проверки в CI
- готовить сводку для проверки pull request
- оценивать рискованные или большие наборы изменений перед правками
- сначала получить замечания, а правки отложить на потом

Ожидаемый результат:

- замечания, сгруппированные по `Reuse`, `Quality` и `Efficiency`
- отдельно отмеченные рискованные пункты
- конкретные структурные рекомендации
- отсутствие изменений в коде

Это рекомендуемая точка интеграции для CI.

### 2. Fix

В режиме `fix` агенту разрешено редактировать код, но только в рамках ограничений review-lenses.

Используйте этот режим, когда:

- разработчик явно просит очистку или упрощение
- реализация уже работает и теперь требует структурной полировки
- проверка выявила безопасные улучшения без изменения поведения
- разработчик хочет, чтобы агент применил минимальный цельный патч, а не только сообщил о проблемах

Ожидаемый результат:

- точечный патч
- сохранённое поведение и публичные контракты
- короткое описание структурных улучшений

Этот режим обычно должен запускаться вручную, а не автоматически в CI.

## Типовые сценарии разработки

### После первичной реализации

Разработчик сначала добивается рабочего поведения, затем запускает `review-lenses` в режиме `fix`, чтобы уменьшить дублирование, упростить поток выполнения и убрать шум.

Это один из самых ценных сценариев применения.

### Перед Commit или PR

Разработчик или проверяющий запускает `review-lenses` в режиме `review-only` на текущем наборе изменений, чтобы получить структурированный повторный проход перед публикацией.

Это особенно полезно, когда код, скорее всего, корректен, но его структура всё ещё слаба.

### Во время review шумного diff

Используйте `review-only`, когда набор изменений трудно читать из-за повторяющихся вспомогательных функций, слишком императивной логики, отладочных остатков или долга по очистке, смешанного с основной задачей.

Цель здесь — получить точные рекомендации до того, как принимать решение о переписывании кода.

### Ручной проход очистки

Используйте `fix`, когда пользователь явно просит:

- упрощение
- очистка
- устранение дублирования
- структурный рефакторинг без изменения поведения

Это предпочтительный путь ручной активации для Codex и Cursor.

### CI Review Gate

Используйте `review-only` в CI, когда нужен:

- структурированный артефакт проверки агентом
- вывод замечаний в PR или журнал сборки
- неблокирующий или блокирующий сигнал качества в зависимости от политики команды

CI должен сообщать, а не тихо переписывать код.

## Модель активации

`review-lenses` — это правило работы, а не автономный фоновый процесс.

Он становится активным только тогда, когда один из следующих механизмов делает его активным в текущем инструменте:

- сгенерированный `AGENTS.md` включает feature `review-lenses`
- локальный managed или custom adapter навыка/правила реализует этот порядок работы
- текущее сообщение в чате явно просит выполнить проход review-lenses

На практике активацию следует считать явной, если только инструмент специально не настроен на автоматическое подключение этого порядка работы.

## Активация в конкретных инструментах

### Codex

Для Codex типовой шаблон такой:

1. Подключить `review-lenses` в стандартах проекта, чтобы правило попало в сгенерированные инструкции.
2. Объявить `codex` в `[tooling].agents` или добавить другой локальный артефакт порядка работы, если у вашей конфигурации свой layout.
3. Запускать этот порядок работы явным запросом в чате.

Типичные запросы:

- `Run a review-lenses pass on my current changes in review-only mode.`
- `Review the changed files through Reuse, Quality, and Efficiency. Preserve public APIs.`
- `Run simplify-review in fix mode and apply only safe cleanup changes.`

Практические рекомендации:

- по умолчанию использовать явную активацию
- применять `review-only` для CI и осторожной проверки
- использовать `fix` вручную, когда разработчик хочет именно правки

### Cursor

Для Cursor типовой шаблон такой:

1. Включить `review-lenses` в сгенерированные инструкции проекта.
2. Объявить `cursor` в `[tooling].agents` или добавить правило проекта в `.cursor/rules/`, например `simplify-review.mdc`.
3. Решить, должно ли правило подключаться вручную или более автоматически, исходя из его типа и области действия.

Типичные запросы:

- `Use the simplify-review rule in review-only mode for this diff.`
- `Use the simplify-review rule and propose the smallest safe fix patch.`

Практические рекомендации:

- сначала предпочитать явный вызов правила
- использовать автоматическое подключение только когда команда уверена, что область действия узкая, а поведение стабильно
- оставлять `fix` ручным, если только нет очень контролируемой схемы автоматизации

### Claude Code

Claude может использовать ту же методику через локальный навык вроде `.claude/skills/simplify-review/SKILL.md`.

Шаблон использования тот же:

- `review-only` для анализа и отчётности
- `fix` для явного ручного cleanup

Claude здесь не особенный; это просто ещё один способ переноса того же порядка работы.

## Автоматическая и явная активация

Как базовая политика команды:

- `review-only` можно автоматизировать в CI или автоматически подключать в узко ограниченных сценариях
- `fix` должен оставаться явным и запускаться по инициативе разработчика

Почему:

- `review-only` безопасен, потому что только сообщает замечания
- `fix` меняет код и поэтому должен иметь явное человеческое намерение

Рекомендуемое значение по умолчанию:

- локальный чат: явная активация
- CI: автоматизированный `review-only`
- автоматический `fix` на весь репозиторий: избегать

## Рекомендации по интеграции с CI

`review-lenses` лучше всего встраивается в CI в режиме `review-only`.

Рекомендуемые обязанности CI:

- запускать детерминированные проверки вроде линтеров, проверки типов и тестов
- запускать проверку агентом в режиме `review-only`
- публиковать замечания в выводе сборки, артефактах или комментариях к pull request

Что CI не должен делать:

- тихо переписывать код
- автоматически коммитить рефакторинг
- запускать широкую автономную очистку без человеческой проверки

Хорошие паттерны для CI:

- неблокирующий отчёт агента на раннем этапе внедрения
- переход к блокировке только для выбранных классов замечаний после стабилизации порядка работы
- явное разделение детерминированных порогов качества и вывода агентной проверки

## Рекомендации по Git hooks

Git hooks могут быть полезны, но workflow нужно разделять аккуратно.

Рекомендуется:

- использовать `pre-commit` или `pre-push` для детерминированных проверок
- добавлять необязательные вспомогательные команды, которые разработчик запускает вручную перед push

С хуками, запускающими агента, нужна осторожность:

- агент может работать медленно
- результаты могут быть недетерминированными
- автоматические правки во время push трудно доверять и трудно отлаживать

Рекомендуемая политика:

- не запускать `fix` из push hook
- использовать hooks только для напоминаний, подготовки контекста или детерминированной проверки
- если AI участвует рядом с push, предпочитать `review-only`

## Предлагаемое внедрение по этапам

### Phase 1

- включить возможность `review-lenses` в стандарты проекта
- задокументировать `review-only` и `fix`
- использовать ручную активацию в чатах

### Phase 2

- добавить managed или custom локальные артефакты инструментов, например навыки Codex или правила Cursor
- стандартизовать формулировки запросов внутри команды
- использовать `review-only` на выбранных PR

### Phase 3

- интегрировать `review-only` в CI
- автоматически публиковать замечания
- решить, какие замечания могут стать блокирующими

### Phase 4

- уточнять эвристики, зависящие от стека, в локальных правилах или стековых фрагментах
- оставлять `fix` ручным, если только команда не построила очень контролируемую схему автоматизации

## Стартовые запросы

Используйте эти запросы как базовый язык команды.

### Review-Only

- `Run a review-lenses pass on my current changes in review-only mode. Report findings under Reuse, Quality, and Efficiency.`
- `Review this diff in review-only mode. Preserve public APIs and highlight structural gains that would come from a safe cleanup pass.`

### Fix

- `Run simplify-review in fix mode on the changed files. Apply only safe behavior-preserving cleanup.`
- `Use the review-lenses workflow in fix mode. If a change is risky, report it instead of editing.`

## Связь с другими механизмами качества

`review-lenses` должен дополнять, а не заменять:

- обычную проверку человеком
- детерминированные линтеры и проверку типов
- тесты
- архитектурные правила и правила обработки ошибок, уже определённые в `ai-standards`

Используйте его тогда, когда обычная проверка слишком расплывчата, но полноценное перепроектирование избыточно.
