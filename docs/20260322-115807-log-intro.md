[CONPORT_ACTIVE] Рабочий вариант здесь не `symlink` как конечное решение, а `central repo + profiles + renderer`, который генерирует `AGENTS.md` в проект по манифесту. Это снимает проблему ручной синхронизации и нормально поддерживает различия по стеку.

Предлагаю такую схему:

1. Завести отдельный репозиторий, например `workspace/ai-standards`.
2. Хранить там не один монолитный `AGENTS.md`, а фрагменты:
   - `core/` — ваши неизменные правила: архитектура, error handling, DRY, git workflow.
   - `process/` — `design-first-collaboration`.
   - `knowledge/` — правила по GRACE.
   - `stacks/` — `python.md`, `node.md`, `fastapi.md`, `django.md`, `react.md` и т.д.
   - `tools/` — Codex/Claude/OpenAI-specific куски, если нужно.
3. В каждом проекте держать маленький манифест, например `ai.project.toml`:
```toml
profile = "uma2"
stacks = ["python", "fastapi", "postgres"]
features = ["design-first-collaboration", "grace", "conport"]
local_overrides = ["docs/ai/project-rules.md"]
version = "2026.03"
```
4. Командой вроде `uv run ai-sync render` собирать из этого итоговый `AGENTS.md` в корне проекта.
5. В заголовок сгенерированного файла писать источник и версию:
```md
<!-- Generated from ai-standards@2026.03. Do not edit manually. -->
```

Почему это лучше текущего симлинка:
- симлинк не умеет стековые различия;
- симлинк сразу меняет поведение всех проектов, без pinning;
- генерация позволяет обновлять проект только когда вы сами поднимете `version` или запустите `sync`;
- итоговый `AGENTS.md` остаётся обычным файлом в корне проекта, без неявной магии для инструмента.

Ключевая мысль: `AGENTS.md` должен быть тонким runtime-артефактом, а не главным местом хранения знаний. Сами знания лучше разнести так:
- `AGENTS.md` — правила поведения агента и ссылки на обязательные артефакты.
- `docs/ai/design-first-collaboration.md` — процесс взаимодействия.
- `docs/ai/grace.md` — как именно применять GRACE к знаниям о кодовой базе.
- ConPort — текущее состояние, решения, прогресс, glossary.

Для GRACE я бы не пытался запихнуть всю методологию внутрь `AGENTS.md`. Лучше так:
- в `core/process` зафиксировать правило: агент обязан использовать GRACE при анализе кодовой базы;
- саму структуру GRACE вынести в отдельный шаблон-документ;
- при инициализации проекта класть шаблон, например `docs/ai/grace-map.md`, и/или заводить категории в ConPort под те же сущности.

Практически я бы сделал именно такой layout:

```text
    core/
    process/
    knowledge/
    stacks/
    tools/
  templates/
    project_manifest.toml

```

А в проекте:
```text

```

Если нужен управляемый rollout обновлений, добавьте pin на commit SHA вместо даты. Тогда обновление выглядит так:
1. меняете центральные правила в `ai-standards`;
2. в нужном проекте обновляете `version`/SHA;
3. запускаете `render`;
4. смотрите diff итогового `AGENTS.md`.

Отдельно: я бы не делал `AGENTS.md` вручную редактируемым вообще. Иначе снова получите дрейф. Лучше жёстко: “generated, do not edit manually”, а все проектные особенности держать либо в `ai.project.toml`, либо в `docs/ai/project-rules.md`.

**1. Project-specific правила**
Это уже почти решено текущей схемой через `local_overrides` в `ai.project.toml`.

Рабочий flow:
- общие правила живут в `~/workspace/ai-standards`
- правила конкретного проекта живут в самом проекте, например:
  - `docs/ai/project-rules.md`
  - `docs/ai/stack-rules.md`
  - `docs/ai/domain-rules.md`
- в `ai.project.toml` они перечисляются в `local_overrides` в порядке приоритета
- `render` встраивает их в итоговый `AGENTS.md`

То есть специфичные правила не попадают в `ai-standards`, но гарантированно исполняются в этом проекте.

Я бы формализовал 3 класса overrides:
- `shared central`: reusable, идут в `ai-standards`
- `project local`: нужны всем в одном проекте, хранятся в репозитории проекта
- `private local`: нужны только вам локально, не коммитятся

Для этого стоит добавить такой flow:
- `docs/ai/project-rules.md` — versioned, для всей команды
- `docs/ai/private-rules.local.md` — optional, в `.gitignore`
- в манифесте:
```toml
local_overrides = [
  "docs/ai/project-rules.md",
  "docs/ai/private-rules.local.md",
]
```
- `ai_sync.py` лучше научить опционально пропускать отсутствующие `.local.md`, если путь помечен как optional

Это даст хороший контракт:
- если правило относится только к одному проекту, оно не должно попадать в `ai-standards`
- если правило относится к нескольким проектам или стеку, его надо поднимать в `ai-standards`

**2. Заимствование внешних правил в `ai-standards`**
Здесь нужен не “copy-paste flow”, а “normalize-and-adopt flow”.
